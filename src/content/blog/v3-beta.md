本日、EQMonitor v3 Beta Programと今後のEQMonitorのリリーススケジュールについて共有できることを嬉しく思います。

GitHub Releasesにて最初のリリースを行ってから4年2ヶ月(2022/04/05)、App StoreとPlay Storeにて一般公開をはじめてから3年半(2023/12/01)が経過しました。開発を始めた当初は高校生だった私もEQMonitorのユーザ数が増えるにつれて歳を重ね、今は一般企業でエンジニアとして勤務しています。

4年間もの間、EQMonitorに対して向き合い続けられたことは、アプリ利用者である皆様のお陰だと思っております。Twitter(X)上でのコメントやメールでの連絡、ストアレビュー等すべて読ませて頂き、開発の大きな励みになっております。本当にありがとうございます

次のメジャーリリースにおいてどのような機能を展開するかと今後の展望についてお伝えできればと思います。

なお、本記事の制作に関わりAIが生成した文章は一切なく、記事のレビューにのみ利用したことを宣言します。 (個人の思想ですが、AIが書いた文章はあまり好きではないのです…)

## 開発者の近況

現在安定版として公開しているEQMonitor v2.6.1のリリース(2024/09/11)から、私の身の回りの環境は大きな変化を迎えました。

## EQMonitor v3で新たに実装/修正した機能

### アプリ側の変更

- 全体
  - アプリケーションのデザインを見直し、細かい配色やデザインを変更しました
  - アプリケーションのフォントを変更しました
- 強震モニタ
  - 地表のリアルタイム震度以外の表示
    - 地表・地中の切り替え, データの種類を震度, 最大加速度, 最大速度などから選べるようにしました
- 緊急地震速報 (表示)
  - 現在地の地域(気象庁の細分化地域)で予想される震度, 到達までの時間を表示
- 地震履歴
  - 地震履歴(2020/11/19~現在)と震度データベース(1919/01/01~2020/11/18)を結合し、1919年から現在までのすべての地震を検索できるようにしました
  - 検索条件に以下を追加しました
    - 地震情報の種別: 地震, 遠地地震, 火山噴火
    - 最大長周期地震動階級
    - 電文種別: 気象庁が発表した電文種別による絞り込み
    - 震源の緯度経度と震源地名
  - **地域の震度で絞り込み**
    - 任意の都道府県もしくは市区町村について、観測した震度で絞り込み
  - 現在地の 市区町村で観測した震度の表示: ホーム画面で利用できます
  - 日毎の区切り
  - その他細かい表示の修正や改善
-
- 通知
  - 緊急地震速報
  - 地震情報
- Live Activity (iOS Only)

### サーバ側の変更

EQMonitor v2までのサーバは単一障害点が多く、サービスプロバイダーの障害に巻き込まれAPIの動作が停止することがよくありました。また、スケールしにくい構成になっており、高負荷時にサーバが応答を停止するケースが多々ありました。

また、パブリッククラウドを主に利用しており収益化を主軸においていないEQMonitorでは大赤字になっていました。

そこで、**EQMonitor v3では、すべてのサーバ側実装を再実装しすべての実装を一から見直しました**

- システム全体
  - システム全体をdocker-composeからkubernetesに変更しました
    - 今まで、APIやサービスの更新の時に数秒落ちてしまうことが
- データベース
  - データベースは3つ用意されており、それぞれ用途に合わせて利用しています
  - PostgreSQL: 地震履歴やユーザの設定値を保持するRDB
    - 今まで、Supabaseを利用して地震履歴やユーザのTokenなどを保持していましたが、自前のPostgresにて運用することにしました
      - Supabaseは時々障害が発生し、単一障害点になっていたためです。また、物理的に離れた場所にデータベースが存在するためパフォーマンスが出にくいことも問題でした。(通知配信のためのデータ取得に平均1,100ms程度かかっていました)
    - CloudNative Postgres(CNPG)を利用して構築しています
      - Read Replica, Multi Region Replicationを構築することも可能なため、v3の安定版リリースまでの間にここらへんを整理・構築する予定です
  - **Valkey (Redis互換): 通知配信最適化のためのKVストア**
    - 配信最適化(1度送ったデバイスに同じ通知を配信しない, 1度送ったデバイスへは後続の情報を配信する)などで利用しています
  - **ClickHouse: アクセス解析や通知配信パフォーマンス監視のためのOLAP (オンライン分析処理)**
    - APIのアクセスやアプリケーションのテレメトリデータを格納し、今後の開発方針の設定や重い処理の発見に利用します
    - サーバの通知配信基盤のデータを格納し、それぞれのイベントの通知配信成功率や遅延を観測します
    - 類似地震の検索にも利用します

- API (eqapi, eqmonitor-websocket)
    -

- 通知配信基盤 (notification-resolver, notification-sender)
  - notification-resolverは、通知を配信するべきデバイスと通知配信Tokenの抽出・それぞれのデバイスの通知メッセージ生成を担います
    - TypeScriptで構築されています
  - notification-senderは、実際の通知配信を担い、FCMとAPNsとの通信を担います
    - Golangで構築されています
    - これまでは、サードパーティのOSSを利用してこの部分を運用していましたが、大量のデバイスへの通知を配信する時にCPUが100%に張り付いてしまい遅延が発生していました
      - そこで、完全自前実装にすることでパフォーマンスの調査や可観測性を担保しやすくしました
- 推計震度データの分析と処理 (ixac41-parser: Rust, TypeScript)
  - 気象庁が発表するIXAC41(250mメッシュ推計震度データ)のデータを処理し、アプリケーションの表示に適した形式(PMTiles)に変換する処理を担います
  - それぞれの処理を並列実行するよう最適化したことで、大きな地震の推計震度でも高速に変換処理を終えられるようにしました
